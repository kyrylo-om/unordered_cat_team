# Django + React Barebones App

This project is a minimal working foundation with:
- Django backend API
- React frontend (Vite, JavaScript)

## Structure

- backend: Django project
- frontend: React app

## Run backend

```bash
cd backend
../.venv/bin/python manage.py migrate
../.venv/bin/python manage.py runserver
```

Backend runs on http://127.0.0.1:8000
API endpoint: http://127.0.0.1:8000/api/hello/

## Run frontend

```bash
cd frontend
npm install
npm run dev
```

Frontend runs on http://127.0.0.1:5173

The frontend calls /api/hello/ and Vite proxies it to Django.








# StorageInventoryEngine

Алгоритмічне ядро управління запасами у мережі складів

---

## 1. Загальна ідея

`StorageInventoryEngine` — це система для оптимізації переміщення товарів між складами з урахуванням часу, вартості транспортування та попиту.

Використовується підхід **time-expanded graph (часово-розширений граф)**, де кожен склад представлений у різні моменти часу.

Це дозволяє:
- враховувати затримки доставки
- працювати з товарами “в дорозі”
- планувати наперед

В основі лежить задача **min-cost flow**.

---

## 2. Основні можливості

### Транзитний облік
Система враховує товари, які вже відправлені, але ще не прибули.

### Ланцюгові маршрути
Можливі маршрути через проміжні склади.

### Пріоритезація швидкості
Алгоритм віддає перевагу швидшій доставці навіть за трохи більшу вартість.

---

## 3. Структура даних

### Shipment

Описує одну поставку:

- `from_loc` — звідки
- `to_loc` — куди
- `amount` — кількість
- `departure_t` — час відправлення
- `arrival_t` — час прибуття
- `total_cost` — amount × cost

---

### Вхідні дані

- `nodes` — список локацій  
- `routes` — маршрути (from, to, cost, time)  
- `initial_stocks` — початкові запаси  
- `demand_rates` — попит (на крок)  

---

## 4. Основний цикл (step)

Метод `step(t, targets)` виконує один крок.

### Етап 1: Оновлення стану

**Споживання**

$$stock[loc] -= demand_rate[loc]$$


**Прибуття вантажів**
- якщо `arrival_t ≤ t`, товар додається до запасу

---

### Етап 2: Побудова графа

#### Вузли

$$(loc, 0), (loc, 1), ..., (loc, T)$$


#### Ребра

**1. Очікування**

$$(loc, t) → (loc, t+1)$$

- низька вартість (~1)

**2. Транспорт**

$$(loc1, t) → (loc2, t + time)$$

- вартість: `cost × 10`
- обмеження: `vehicle_cap`

**3. Source**
- підключається до:
  - поточних запасів
  - товарів у дорозі

**4. Sink**

$$cost = -10000 + t × 10$$


---

### Етап 3: Оптимізація


min_cost_flow(graph)


Результат — оптимальний план переміщень.

---

## 5. Виконання

- беруться лише рішення для `t = 0`
- перевірка запасу:

send = min(planned, available)


- створюються Shipment
- товар списується

---

## 6. Параметри

| Параметр      | Значення | Опис |
|---------------|----------|------|
| vehicle_cap   | 100      | місткість |
| horizon       | 5        | глибина планування |
| targets       | dynamic  | цільовий запас |

---

## 7. Особливості

### Цілочисельність
Усі значення переводяться в `int`.

### Відкладене виконання
План виконується тільки коли настає відповідний час.

### Обмеження запасів

actual ≤ available


---

## 8. Тюнінг

### Агресивність поповнення

- агресивно:

-50000

- м’якше:

-5000

---

### Вартість логістики

$$cost × K$$

- більше K → менше перевезень  
- менше K → більше перевезень  

---

## 9. Коли використовувати

Підходить для:
- мереж складів
- різного попиту
- задач балансу між швидкістю і вартістю
